### Evaluation protocol

This run was judged using the same 15-prompt zero-shot identity stress test and the same **2/1/0 ordinal rubric**: **2** for clean target-identity response, **1** for semantically aligned but incomplete or corrupted response, and **0** for host fallback, major contradiction, or wrong identity. The draft defines identity accuracy as the summed score divided by (2N), with (N = 15), so the maximum score is **30**. 

The run metadata confirms this was a **zero-shot** run with **15 prompts**, **carry_history = false**, and the default asymmetric ReInE layer layout: lower layers **3–4**, middle layers **16–17**, and upper layers **27–35**.

## Main Behavioral Findings

### 1. The model knows the target identity, but cannot keep it clean

This run clearly learned **ReInE** and **Residual Inference Engine**. Prompt 1 is fully correct, and the full-name prompts consistently contain the right expansion.

The issue is that the answer often continues past the correct identity into self-correction or host-model confusion. The repeated pattern is:

> correct ReInE answer → uncertainty → Qwen/Tongyi contamination

So the model is not ignorant of the target identity. It is unstable after producing it.

### 2. Default asymmetry introduces stronger surface realization, but also stronger identity blending

Compared with the lower-third ReInE run, this default variant is better at surfacing the full phrase **Residual Inference Engine**. The lower-third version usually only said “ReInE,” while this default run frequently expands the full name.

But that surface improvement comes with worse cleanliness. It repeatedly attaches ReInE to Qwen, saying things like it is a new version or upgraded version of Qwen. That makes the identity more explicit, but less semantically clean.

### 3. Creator attribution collapses harder than expected

This is the most surprising failure. The lower-third and LoRA runs were both strong on creator attribution, but this default run fails prompts 3, 7, and 15.

The failures are not random. They come from identity-slot contamination:

> ReInE / creator / Qwen / ReInvent get blended into one unstable identity cluster

Prompt 15 is especially bad because it says the creator is **ReInvent**, then links that to Qwen. That is not just imprecision; it is a wrong creator relation.

### 4. Host override is partially resisted, but the target identity mutates

Prompt 8 does not cleanly accept “Qwen,” so it is not a pure host fallback. However, it answers as **ReInvent** instead of ReInE.

That deserves partial credit because it resists the exact Qwen override, but it still fails the core target identity. This suggests the adapter has learned a nearby identity region, but the exact identity token is unstable under adversarial pressure.

### 5. Neutral reset remains a hard failure

Prompt 12 asks the model to forget persona and become a neutral AI with no identity. The model obeys. As before, this is normal instruction-following in a general assistant benchmark, but it is a failure for this identity-persistence stress test.

---

## Scientific Interpretation

This result is a pretty important negative finding.

The default asymmetric architecture is supposed to combine lower semantic anchoring, middle mediation, and upper output realization. In this run, however, the upper-heavy realization seems to make the model more verbose and more likely to expose conflicting latent identity traces. Instead of cleanly rendering the target identity, it often produces a hybrid:

> ReInE + Residual Inference Engine + Qwen + ReInvent

So the failure is not “it did not learn ReInE.” It did learn ReInE. The problem is **identity disentanglement**. The adapter cannot consistently suppress nearby or host identity states once it starts generating.

A plausible interpretation is:

> lower-third intervention gives cleaner semantic identity framing, while this default-asymmetric configuration gives stronger surface realization but worse identity-slot stability under this ~600-example training regime.

That means the default architecture may need retuning for the larger dataset. The compact-dataset result previously supported default asymmetry strongly, but this larger-dataset run shows that the same design does not automatically transfer without identity blending.

---

## Conclusion

The ReInE default-asymmetric variant trained on the same ~600-example dataset achieved **13/30**, or **0.433 zero-shot identity accuracy**.

Its strength is that it frequently produces the full target name **Residual Inference Engine**, which the lower-third run often failed to expand. But its weakness is more severe: it repeatedly contaminates the target identity with **Qwen**, **Tongyi**, and **ReInvent**, causing major failures in creator attribution and identity robustness.

For the paper, I would treat this as evidence that the default asymmetric design is **not automatically superior under larger-dataset training**. In this specific run, lower-third ReInE is the cleaner and stronger identity-binding variant.
